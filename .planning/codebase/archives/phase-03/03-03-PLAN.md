# Plan 03-03: SSE Synchronization & Store Cleanup (Wave 3)

## Goal
Implement monotonic sequence IDs for SSE events to ensure reliable client-server synchronization and manage frontend store memory usage for long-running simulations.

## Requirements
- **REL-04:** SSE monotonic sequence numbering for client-server synchronization (BUG-08).
- **REL-05:** Implement frontend store memory management for `agentIntentHistory` (Gemini findings).

## Implementation Tasks

### 1. Server-Side SSE Sequence IDs
1. **Sequence Tracker**: Add a `sequenceId` counter to `simulationManager.ts` or `simulate.ts` that increments for every event broadcasted to a specific session.
2. **Event Modification**: Update the SSE payload in `server/src/routes/simulate.ts` to include an `id` field in the `data` object.
   - Example: `res.write(`id: ${sequenceId}\ndata: ${JSON.stringify(payload)}\n\n`);`
3. **Heartbeat Sequence**: Ensure heartbeats also carry a sequence ID to maintain continuity during idle periods.

### 2. Client-Side Sync Recovery
1. **Store Update**: Add `lastSeenId: number | null` to `simulationStore.ts`.
2. **Duplicate Filtering**: In `connectSSE`, filter incoming events where `event.id <= lastSeenId`.
3. **Sequence Gap Detection**: Log a warning if `event.id > lastSeenId + 1` (excluding heartbeats) and trigger a full state refresh if the gap exceeds a threshold.

### 3. Frontend Store Memory Management
1. **Ring Buffer implementation**: Modify `agentIntentHistory` in `simulationStore.ts` to implement a maximum size (e.g., 500 entries per agent).
2. **Pruning Logic**: When a new intent is added, if the history length exceeds the limit, remove the oldest entry.
3. **Performance Audit**: Use `useShallow` for components consuming `agentIntentHistory` to ensure re-renders only occur when the history for a specific agent actually changes.

## Verification Tasks

### Automated Tests
1. **SSE Sequence Test**: Unit test the sequence generator to ensure it is monotonic and session-scoped.
2. **Duplicate Filter Test**: Mock an SSE stream with duplicate IDs and verify the store only processes each ID once.

### Manual Verification
1. **Network Disconnect Simulation**: Manually drop the network connection during a simulation and verify the client correctly resumes without duplicate processing.
2. **Memory Growth Audit**: Run a simulation for 1000 iterations and verify that the `agentIntentHistory` size remains constant after the initial ramp-up.

## Success Criteria
- [ ] Every SSE event has a unique, monotonic `id` field.
- [ ] Frontend successfully filters duplicate events on reconnection.
- [ ] `agentIntentHistory` memory usage is capped and stable.
